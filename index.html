展开看原理 ↓

一句话:本工具做的事等价于 Codex CLI 官方的 codex login --with-access-token 命令(见 OpenAI 官方文档),只是把它做成了网页版。

  1. ChatGPT 是怎么登录的:你在浏览器里手动登录,经过密码 + 验证码 + 2FA 等,OpenAI 会颁发一个 access token(JWT 格式,带签名 + 有效期 + 权限)给浏览器。浏览器把 token 存到 cookie 里,后续每次请求都会带上。
  2. 这个 token 是什么:它本质是你的身份证明。只要没过期,任何持有这个 token 的程序都可以用你的身份调用 OpenAI API。JWT 是 OAuth 2.0 标准,OpenAI 服务器端每次 API 请求都会验签 + 查有效期 + 查权限。
  3. 本工具做的事:浏览器地址 chatgpt.com/api/auth/session 会返回当前会话的 token JSON。本工具读这个 JSON → 抽出 accessToken → 包成 Codex CLI 认识的 ~/.codex/auth.json 格式 → 给你生成写入命令。
  4. "免验证码"是免在哪:验证码是 ChatGPT 登录流程里防机器人用的;你已经手动登录过了,拿到的 token 就是合法身份。Codex 拿到这个 token 后调用 API,OpenAI 那边只认 token 不认"是不是浏览器",所以不需要再走 OAuth 流程。
  5. 安全保证:整个过程在你浏览器本地完成,token 不会上传到任何服务器(本工具是纯前端 HTML,没有后端)。

出处:openai/codex 源码 auth/storage.rs · Codex 官方认证文档

? 常见问题
token 会过期吗?

会。ChatGPT 的 access token 有效期大约 10 天。过期后 Codex 调用 API 会 401,需要重新跑一次本工具拿新 token。token 里的 exp 字段就是过期时间。

为什么我的 auth.json 里 refresh_token 是 "placeholder"?

因为本工具只能从 /api/auth/session 接口拿 accessToken,拿不到真的 refresh_token(OpenAI 没暴露这个)。所以放一个 "placeholder" 占位(不能空字符串,Codex CLI codex doctor 会报 "missing a refresh token")。token 过期后 Codex 无法自动续期,你得重新跑一次。如果想自动续期,需要走 Codex 官方 codex login(浏览器 OAuth)流程。

安全吗?会不会泄露?

本工具是纯前端,数据不离开你的浏览器。但 token 本身是敏感信息 —— 任何拿到这个 token 的人(到过期为止)都能以你身份调用 OpenAI API。不要发到公网(GitHub、Discord、博客等),不要截图发群里。本地存也建议 chmod 600 ~/.codex/auth.json。

为什么不直接用 codex login ?

官方 codex login 要打开浏览器走 OAuth 回调。在以下场景会失败:

  • 无 GUI 的服务器 / SSH 远程
  • 浏览器 localhost 回调被防火墙拦了
  • 自动化脚本 / CI/CD runner
  • 不想每次重新登录(token 还可以用)
本工具就是给这些场景准备的等价方案。

为什么不用 API key?有 API key 不就行了?

API key 是 OpenAI 开发者付费,按 token 用量扣钱,跟 ChatGPT Plus 月费订阅是两套独立计费。如果你已经有 ChatGPT Plus/Pro/Team 订阅,用本工具能直接复用订阅额度,不用另外充值 API 余额。

可以把这个 auth.json 复制给朋友用吗?

技术上能跑,但 不建议。原因:

  • refresh_token 是单次使用,谁先刷谁生效,其他人会突然登出
  • 从 OpenAI 服务条款看,分享账号凭证属于违规
  • token 泄露后无法远程撤销(只能等 10 天过期)
